① 瞬断後の障害対応（東電瞬断）

何が起きていたか

東電の瞬断により踏み台サーバー（Bastion VM）が停止し、EMSに接続できなくなった。基盤電源断後の正規の起動手順が実施されていなかったことが原因。

チャットの流れを追うと

  • 「Bastion VMが停止している状況を確認」 — 踏み台が落ちたのでEMSに辿り着けない。
  • 「正規の起動手順が実施されていないように見受けられる」 — 手順書通りの電源ON作業が必要。
  • 「再開ツールでEMS Podを落として再度起動を」 — EMS Podの再起動で復旧を試みる。
  • 「4G EMSでCell ListやLink Listを表示しようとするとError」 — EMS接続できてもアプリ層でエラーが出る場合がある。
  • 「5G EMSは問題なし」 — 影響範囲は環境によって異なる。
  • 「kubectl namespace指定でForbiddenエラー」 — NSなしで実行すればoc loginしているNSのPodは表示される。
  • 「イベントリストが正常に出力、NE connection lost ALMが解消」 — 復旧完了の判断。

復旧の判断基準

確認項目正常の判断
EMS接続可能
リンク状態各種リンクOK、ALMなし
イベントリスト最新日付で更新されている
Pod状態正常
NE connection lost ALM解消
🔑 学び：Bastion VMが止まると全てに影響する。UPS（無停電電源装置）の有無が重要。kubectl namespace指定でForbiddenが出てもNSなしで確認可能。

② kubeconfig取得トラブル（CU認証NG）

何が起きていたか

インスタ・1callまで成功したが、CU側のAdminMachineで認証NGエラーが発生し、コマンドが打てなくなった。kubeconfigファイルが古く、再取得しようとしたらAWS S3からForbiddenエラー。root権限で実行して解決。

チャットの流れを追うと

  • 「インスタOK＆1callも通った」 — ここまでは正常。
  • 「CU側はAdminMachineで認証NGっぽいエラー」 — kubeconfigが古いか権限の問題。
  • 「EMSでconnectになるので、CU-CPをインスタしてみた」 — EMS上での操作で状態確認。
  • 「kubectlのconfigファイルを新しくしないとかも」 — kubeconfigの更新が必要。
  • 「O-cloudポータルにDL方法ある」 — 手順はO-cloudポータルに記載。
  • 「DU-1にs-plane manager SW failureのAlarm」 — DUインスタ後にアラーム発生。
  • 「fatal error、the Head object operation Forbidden」 — AWS S3からアクセス拒否。
  • 「root権限で実行したらできました」 — sudo -i → chmod 600 で解決。

解決手順

手順コマンド
1. rootに切り替えsudo -i
2. 環境変数を設定（O-cloudポータル参照）
3. kubeconfig権限変更chmod 600 [kubeconfigファイル名]
4. rootを抜けるexit
🔑 学び：kubeconfigが古いと認証エラーになる。S3からのDLでForbiddenが出たらroot権限を試す。参照：o-cloud保守マニュアル-r2.0 6.3.2

③ F-DU Fronthaul同期エラー

何が起きていたか

富士通のDU（F-DU）でFronthaul-synchronization Errorアラームが発生。PTPのClockclass上限値(7)を超えた値(180)がアナウンスされたことが原因。RUの全キャリアがDisabledになりセルもLockedになった。他系でも同時刻に発生していたため、上流の時刻源の問題と推測。

チャットの流れを追うと

  • 「F-DUでFronthaul-synchronization Errorが発生」 — DU-RU間の時刻同期が失敗。
  • 「RU All TRX Array Carrier Disabled、sync-state:UNLOCKED」 — RUの全キャリアが無効化。
  • 「他系でも同じ時間に同様のalarm」 — 特定の系だけの問題ではない。上流が原因。
  • 「EMSの画面が確認できるので試験観点は満たせそう」 — アラーム中でもEMSで確認できれば試験は進められる。
  • 「Clockclassの上限値(7)を超えた値(180)がアナウンスされた」 — 障害ログから原因特定。
  • 「Alarmはクリアになってました」 — 自然解消。早めに対応された可能性あり。

障害ログ例

stream 0: maxclockclass(7) exceeded by Rx Announce Class(187)
→ Clockclassの上限値7を超えた値がアナウンスされたため同期エラー
🔑 学び：S-Plane（PTP）の問題。Clockclassは値が小さいほど高品質、上限値7を超えると同期NG。他系でも同時発生なら上流の時刻源を疑う。アラーム中でもEMS確認できれば試験を進められる場合がある。

④ NGリンク接続トラブル（IPアドレス設定不一致）

何が起きていたか

CUをリセットしたがAMF（コア側）とのNGリンクが繋がらなかった。原因はIPアドレスの設定が合っていなかったこと。Wiresharkで確認してSCTP initが飛んできていることを発見し、AMF SIMを立ち上げて解決。

チャットの流れを追うと

  • 「NFリセットしてもNGリンク繋がらない」 — CUのPodを再起動したが、AMFとのNG接続（NGAP）が確立しない。
  • 「アドレス変えないとダメ」 — IPアドレスの設定が基盤側と合っていない。ここでネットワーク知識（サブネット /27 や /23）が必要。
  • 「AMF側が/23なのにarpできない」 — 同じネットワークに見えるはずなのにARP（MACアドレス解決）ができない。GWの設定が原因。
  • 「局データも直さないと」 — IPアドレス変更に伴い局データも修正が必要。
  • 「基盤更改は1.32の状態（切り戻し相当）で戻ってくる」 — もう一度CUアップグレードが必要。
  • 「wiresharkみてた。SCTPのinit飛んできてた」 — ここが決め手！ WiresharkでSCTPの接続開始パケットを確認して原因を特定。
  • 「AMF SIM立ち上げたらつながった」 — AMFシミュレータを起動したらSCTPが繋がった。
  • 「NSもLabelもない」 — 基盤再構築でKubernetes上のnamespace/Labelが初期化されていた。
  • 「F1が繋がってMplaneも見えた」 — F1（CU-DU間）とM-Plane（管理）が両方確認できたので復旧完了。
🔑 学び：
• ping、ARP → ネットワーク基礎
• SCTP → F1-CやNGAPの制御プロトコル
• Wireshark → SCTPのinitパケットを見て判断（この場面で決め手になった）
• kubectl → Pod状態確認、インスタ
• F1、M-Plane → インターフェースとプレーン

pingが通らない → アドレスを変えてみる → まだダメ → Wiresharkで見たらSCTP initが飛んできていた → AMF SIMを立ち上げたら解決。Wiresharkがなければ「何が起きているか」がわからないままだった。

⑤ Rel4.5 CU-CP NGリンク不確立（局データ読み込み失敗）

何が起きていたか

VLAN850へ移行後、Rel4.5のCU-CPを立ち上げたがAMFとのNGリンクが確立しなかった。アドレス・ポート・バージョン差分を一通り疑ったが、最終的な原因は局データがEMSに正しく読み込めていなかったこと。EMSからresetして読み込み直し、最新版(R04.50.20)で再インスタしたらNG・E1ともに繋がり、2セルACTIVEまで到達した。

チャットの流れを追うと

  • 「Rel4.5のCU-CP立ち上げたがNGリンク確立しない」 — VLAN850移行の影響を疑う。
  • 「局データは変えてない、NGリンクのIPアドレスも合ってそう」 — まず設定値を確認。AMF-SIMが立ち上がっているか担当が確認。
  • 「ログに 172.16.40.36:50052 宛のエラーが繰り返し」 — アドレスかポート違いを疑う。
  • 「前の版数(ターミ直前)まではAMFと通信できていた」 — 今回版とリクエストパラメータの差分確認 → imageのバージョン以外差分なし。
  • 「前の版でインスタしてもNGリンク繋がらず、E1もDisabledのまま」 — E1も繋がらない＝局データかインスタパラメータが怪しい。
  • 「N2podのアドレスは 10.48.52.16 で合ってる」 — パラメータは問題なし。
  • 「EMSには繋がっている。局データ読み込めているか? configmgr のログをzipでgrepして局データファイルを確認」 — ここが切り分けの肝。
  • 「局データ読めてないですね。EMSからreset」 — リセット実行。
  • 「読み込みました。EMSのリンクリスト上にNGリンクも出てきた。NG・E1繋がった」 — reset後に解消。
  • 「R04.50.20(最新版)で再インスタ → NGリンク接続成功」 — CU-CP単独でNGリンクが繋がる瞬間を初めて観測。
  • 「UP4までインスタ → DUインスタ → 2セルともACTIVE」 — 復旧完了。
  • 「DUをターミしてRU増設を試行。F45(一体型)を後接続するためケーブルを抜く → EMS上Disconnect → 再インスタ → SS34セルAct → F45接続 → F45もセルAct」 — 増設も成功。

上流から1つずつ潰していくのが基本。今回は最後の「局データ読み込み」で当たった。

前段：IP取得トラブル（VLAN誤接続）

💡 この日の前段で、VMに繋がらずIP再取得を試みる場面があった。/release → 開発LANのNIC抜く → 1分待つ → 繋ぎ直す → ipconfig 確認、でもIPが変わらず。見立ては「本来 172.21.98.x が割り当たるべきところ、VLAN100側のDHCPに繋がって 172.29.23.x をもらっている」と思われる。時間をおいたらVM接続でき解消。
🔑 学び：
• NGリンクが張れない時の犯人は、アドレス/ポートだけでなく局データの読み込み失敗もある。
• E1もNGも両方Disabledなら、局データorインスタパラメータを疑う(片方だけなら別の切り分け)。
• configmgr のログをgrepすれば局データが読めているか確認できる。
• 困ったら EMSからreset → 再インスタ が効くことがある。
• IPが想定と違う時は、繋いでいるVLANとDHCPの組み合わせを疑う(VLAN100とVLAN850の取り違え)。

OPSログのマーク類 — なぜ出るのか 追加

大前提：N3002の出荷試験OPSログ13本はすべて正常終了(OK)となった完了済みログ。これらのログに出る5つのマークは「OKログに当たり前に出る想定内のマーク」であり、異常の証拠ではない。
🔑 読み方の核心2点
①そのマークの直前に「状態を揺らす操作」があったか（制御/リセット/再開/停止/閉塞）→ あれば「揺らした結果だから正常」、なければ本当の異常かも
②☆コメント行を読む → シナリオが「期待されるNG」を自分で宣言している
マーク意味なぜ出るか判断
受NG 装置が要求を今は受け付けられない 状態変化中に要求を投げたため弾かれた 再送で通れば問題なし
リ断 リンク断 切る操作をした/リセットで一時的に切れた/物理回線を止めると上位リンクも連鎖で切れる リセット・制御の直後なら正常
ALM アラーム報告 意図的にカードを閉塞した/リセット直後で装置がまだ立ち上がり中 閉塞・立ち上がり中なら正常。復旧操作で消える
ERR 対象がエラー状態 閉塞でカードがエラー状態になった 閉塞解除/リセット/復旧で消える（早見表既出「A面指定リセット後に消える」）
処NG 処理失敗 ①停止操作で処理が中断した ②BBカードリセットで発生する既知事象(OAM(N)_s12B_004整理済み) ☆注記に「整理済み」「想定内」があれば正常。注記なしは要確認
💡 ☆注記の実例3つ（シナリオが自ら宣言した期待されるNG）：
「MME6台の場合、最後のMME制御は制御NGとなります。」（シナリオ3）
「BBカードリセット動作となる。OAM(N)_s12B_004にて事象は整理済み」（シナリオ12）
「OF-AMPは非対応のため『制御失敗』となります。」（シナリオ13）

目的：AP(呼処理アプリ)の機能ブロック試験。呼処理に関わる各機能を「制御→通知→復旧」のサイクルで一つずつ確認する。

【リ断】隣接eNBリンクの箇所

隣接eNBリンクを「休止制御」「工事中設定」で意図的に切っています。切る操作をしたのだから切れるのは当然です。復旧操作の後にリンクが戻ることを確認するのがこの試験の目的なので、リ断は「試験が正しく動いている証拠」です。

【ALM】セル設定の箇所

カードを意図的に閉塞しています。「閉塞」とは、そのカードを強制的にアラーム状態にする操作です。ALMが出るのは当然で、閉塞解除後にALMが消えることを確認するのがセットになっています。

【受NG】各所で散発的に発生

制御操作の直後、装置がまだ状態変化中のタイミングで次の要求を投げると弾かれます。「今忙しいから後にして」という装置の返答です。少し待って再送すれば通ります。

時刻偽装＋トラヒック収集（シナリオ後半の伏線回収）

時刻を固定値(2010/08/01)に偽装設定 → 即時トラヒック収集の通知タイムスタンプが偽時刻になることを確認 → 非定期トラヒックは.UENファイル取得 → 時刻を現在時刻へ戻して正常終了。「なぜ時刻を偽装するのか」は、トラヒック収集機能が正しく時刻を記録できるかを確認するためです。偽の時刻を入れて、収集データにその時刻が刻まれるかを検証しています。

目的：APとPFにまたがる共有機能の試験。SDM面・eNBリセット・対MMEリンク・物理回線・Tracerouteを扱う。
流れ：SDM面状態(制御→通知→復旧) → eNBリセット(割り込み処理) → 対MMEリンク(休止→全断→工事中→全復旧) → 物理回線(休止→復旧) → Traceroute導通試験 → 正常終了

【受NG】eNBリセットの箇所で大量発生

eNBリセットをかけると装置が再起動中になります。その最中に状態取得要求を投げても「今リセット中だから受け付けられない」と弾かれます。数分の空白を挟んで再起動が完了すると、その後の要求は通ります。リセット中の受NGは完全に正常です。

【受NG】対MMEリンクでも発生（☆注記の実例）

MME6台を順番に制御していく中で、☆注記に「MME6台の場合、最後のMME制御は制御NGとなります。」と書いてあります。シナリオ自身が「最後の1台はNGになる」と宣言している。これが「☆コメント行を読む」の典型例です。注記があれば想定内。

【リ断】物理回線を止めた箇所

物理回線を休止すると、その上に乗っている対MMEリンク・対隣接eNBリンクも連鎖して切れます。物理層を止めれば上位のリンクも切れるのは当然の依存関係です。「物理→論理」という層の構造がログで見えている瞬間です。物理を復旧すれば上位リンクも戻ります。

【処NG】Traceroute停止の箇所

Traceroute試験を「停止要求」で中断した結果です。処理を途中で止めたので処理が完了しなかった＝処NGという記録が残ります。ただしシナリオとしては「停止できたか」を確認するのが目的なので、これで正常終了しています。チェックリストの「処NG=再試験」は原則ですが、停止操作の結果として出た処NGは例外になります。

シナリオ4「PF固有」約12分

目的：PF(Platform=基盤部)の機能試験。遠隔システムリセットで基盤を実際に再起動し、戻ってくるまでを確認する。さらにOPSファイルの取得サイクルを試験する。ASCIIアート「PF-UNIQUE」。
流れ：遠隔システムリセット → 「10.1.1.6から切断しました」→ 再開通知待ち(残り3分→2分→1分のカウントダウン) → 「10.255.68.11から10.1.1.6へ接続しました」で復帰 → 全状態取得一式 → OPSファイル取得サイクル(.MOA) → 正常終了

【受NG】【ALM】【ERR】再接続直後の全状態取得

遠隔リセットで装置が実際に再起動しています。「10.1.1.6から切断しました」という記録が残るくらい、本当に落ちて戻ってくる。再接続直後は装置がまだ完全に立ち上がりきっていないので、この状態で状態取得要求を投げると受NGが出ます。ALMは「立ち上がり途中のカードがまだアラーム状態」、ERRは「立ち上がり途中のカードがエラー状態」という記録です。時間が経って安定すれば全部消えます。

OPSファイル取得サイクル ★最重要

⚠️ GEN試験PCの流し込み手順に直結する可能性あり（要指導・要確認）

流れ：①使用ファイル(.MOA)を /fm/ftp/put と /fm/log/oam_ap へ転送 → ②OPSファイル取得開始制御 → ③.MOA.gz を /fm/ftp/get へ転送成功 → ④OPSファイル取得終了制御 → ⑤ファイル情報取得 → 削除 → 再度ファイル情報取得(消えたか確認) → 正常終了

つまり「ファイルを置く → OPS経由で取得(get) → 確認 → 削除」という、ファイルの出し入れと後始末が一つの試験になっています。このサイクル部分にはマークは出ません。ファイル操作だけで装置の状態を揺らす操作がないからです。

シナリオ5〜13

シナリオ5〜10「カード診断」の共通構造

シナリオ5〜10は「カード/機能部の個別診断」パート。BDE(基地局を構成する機能部)を一個ずつ自己診断にかけて、全部正常に応答するかを確認する。全て同じテンプレで動いていて、対象カード名だけ差し替わる。各約1分。
🔑 共通テンプレ：カード全状態取得 → カード診断要求 → カード状態変化通知 → カード診断応答 → カード全状態取得 → swait → 正常終了

基本的にマークなし。カード診断は「診断してください」と命令を投げて「異常なし」が返ってくるだけで、カードを閉塞するわけでも、リセットをかけるわけでも、リンクを切るわけでもない。状態を揺らす操作が一切ないのでマークが出る場面がない。診断要求の後に「カード状態変化通知」が来るが、これは「診断中にカードが内部的に自己チェックを走らせた」という正常な報告。

シナリオ5「eNB-CNT」

eNB-CNT(基地局装置監視制御機能部=コントローラ)カードの単体自己診断。約1分。
CNT=CoNTrollerの略。基地局全体の監視・制御を担う「司令塔」のようなカード。無線回線管理や呼処理制御を行う。テンプレそのまま、マークなし。

シナリオ6「TRX-LTE」

TRX-INF(無線送受信インタフェース機能部)のLTE側カードの単体自己診断。約1分。
TRX=Transmitter/Receiverの略。基地局はアンテナで電波を出したり受けたりしているが、そのデジタル↔アナログ変換や変調・復調を処理するのがTRXカード。いわば基地局の「口と耳」にあたる部分。TRXが壊れると電波が出なくなる、または受信できなくなる。テンプレそのまま、マークなし。

シナリオ7「BB-LTE」★マークあり

BB(ベースバンド信号処理)のLTE側カードの診断。約3分。シナリオ5・6と違い、カード制御(閉塞)と装置内試験が追加されるため複雑化する。
流れ：カード全状態取得 → カード診断要求 → 状態変化通知 → 診断応答 → カード全状態取得 → カード制御要求(閉塞) → swait_70(70秒待ち) → 通知大量発生 → カード全状態取得 → 機能部全状態取得 → セル全状態取得 → 装置内試験要求 → 装置内試験応答 → 正常終了

【ALM】カード制御(閉塞)の直後に大量発生

BBカードを意図的に閉塞しています。閉塞＝強制的にアラーム状態にする操作なので、ALMが出るのは当然です。しかもBBカードは上位にあるため、閉塞すると配下のカード(機能部・セル)にも状態変化が波及します。通知に"8HRR2AD""2GLR2AA""BB-LTE""eNB-CNT"といった複数のカード名が登場するのはそのためです。1枚のカードを閉塞すると、関連するカード全体に影響が出るという依存関係がログで見えています。

【ALM】装置内試験の前後でも発生

装置内試験＝カード自身が内部の回路を自己チェックする試験です。この試験中にカードが一時的にアラーム状態になります。試験が終われば復旧します。

swait_70(70秒待ち)の意味

閉塞操作の後、70秒待っています。閉塞によって波及した状態変化が全部落ち着くまで待つための安定化時間です。シナリオ5・6のswait_10(10秒)より長いのは、それだけ影響範囲が広いからです。

シナリオ8「HWY」

HWY-INF(伝送路インタフェース機能部=IP層以下の伝送路を扱う部分)カードの診断。約1分。カード診断テンプレに遠隔EM実施判定が冒頭に追加されている。マークなし。
流れ：遠隔EM実施判定処理(再開状態取得→遠隔EM実施済み確認) → カード全状態取得 → カード診断要求 → 状態変化通知 → 診断応答 → カード全状態取得 → swait_10 → 正常終了
💡 なぜ冒頭に判定が入るのか：EM＝Element Management、つまりEMS(Element Management System)のことです。HWYは伝送路の入口にあたるカードなので、「この装置は遠隔(EMS経由)で管理されているか、それとも直接接続か」を最初に確認しています。管理形態によって次の処理が変わるため、他のカード(eNB-CNT・TRX・BB)にはない判定がHWYにだけ入っています。

シナリオ9「PORT」

PORT(ポート/物理インタフェース)カードの単体自己診断。約1分。テンプレそのまま、マークなし。

PORTカードはイーサネットポートや光ポートなど、外部と繋がる物理的な接続口を管理します。シナリオ3(AP/PF共有)で「物理回線を休止すると対MME・対隣接eNBリンクが連鎖してリ断する」という場面がありましたが、あの「物理回線」を実際に担っているのがこのPORTカードです。シナリオ3のリ断の根本にいたカードを、ここで単体診断にかけています。

シナリオ10「SDM-LTE」

SDM(局データ保持の記憶系カード)のLTE側の単体自己診断。約1分。テンプレそのまま、マークなし。

SDM＝Station Data Memoryの略。基地局の設定情報(局データ)をA面・B面の2面で二重保持しています。前手順で「SDM面A/B等価のpause目視」が毎回入っていたのを覚えていますか。あれは「A面とB面の局データが同じ内容か」を確認する手順で、このSDMカードを使った確認作業です。FLD制御で「A面の局データをB面に転送してA面/B面を同値にする」のも、このSDMカードのA面・B面を操作しています。局データクロスチェックの仕事と直接つながっているカードです。

シナリオ11「新世代RE共通」(CPRI-MUX含む) 約57分

目的：RE(Radio Equipment=無線装置)との光伝送路(CPRI)を含むリセットを繰り返し、無線装置が毎回ちゃんと再接続・立ち上がるかを反復確認する。ASCIIアート「NEO」。
🔑 構造：eNBリセット(CPRI-MUX含む)ゾーンを3回以上繰り返す。
1ゾーン＝リセット要求 → 「10.1.1.6から切断しました」→ 再開通知待ち(残り3分→2分→1分) → 「10.255.68.11から10.1.1.6へ接続しました」→ 全状態取得一式 → State_Acquisition → 「装置立ち上がりまでしばらくお待ちください」→ TILT-INFヘルスチェック通知多数 → Dual/シングル判定 → また次のリセット… → END

【受NG】State_Acquisitionの整流装置・空調機で発生

リセット直後、装置がまだ完全に立ち上がりきっていない状態で状態取得要求を投げています。「今立ち上がり中だから受け付けられない」という返答です。ログにも「装置立ち上がりまでしばらくお待ちください」と書いてある。時間が経てば通ります。

【ALM】架全状態取得の箇所で発生

同じく立ち上がり中のカードがまだアラーム状態にある記録です。完全に起動すれば消えます。「リセット掛かってれば不安定は正常」の典型例。

Dual/シングル判定とは

各ゾーンの締めに必ず入る分岐。局データ情報を取得して「この装置はDual構成(冗長二重化)か、Single構成か」を判定して処理を分岐します。新世代REが二重化に対応しているからこその分岐で、構成によって次にやることが変わります。

なぜ3回以上繰り返すのか

1回リセットして戻ってくるだけでは「たまたま動いた」かもしれない。複数回繰り返して毎回正常に戻ってくることを確認することで、再起動耐性を検証しています。57分かかるのもこの反復のためです。

シナリオ12「新世代RE共通_2」(CPRI-MUX含まない) 約3時間

目的：REとの光伝送路(CPRI)は保ったまま、eNB側だけをリセットする試験。シナリオ11との対比が核心。eNBリセット応答待ち58分×3ゾーンで約3時間になる。

シナリオ11との違い

シナリオ11はCPRI-MUXを含むリセット(光伝送路ごとリセット)で再開まで約3分。シナリオ12はCPRI-MUXを含まないリセット(eNB側だけリセット、光伝送路は保つ)で再開まで58分。

なぜ含まない方が長いかというと、光伝送路が切れないので無線装置側は再起動せずに待ち続けている。eNBが再起動して戻ってきて再同期するまでの時間が長くかかるためです。

【処NG】★最重要の注記実例

リセット応答に【処NG】が出ますが、直前の☆行にこう書いてあります。

「BBカードリセット動作となる。OAM(N)_s12B_004にて事象は整理済み」

これが「☆コメント行を読む」の最も明確な実例です。この処NGは偶発的な異常ではなく、BBカードがリセットされるときに必ず発生する既知の動作で、管理番号(OAM(N)_s12B_004)まで採番されて整理が終わっている。だから出ても問題なし。☆注記に「整理済み」とあれば想定内というルールがここで確定します。

シナリオ13「N-LRE 出荷オペ」約19分

目的：1セクタ・1キャリアに対して実運用操作を一通り流す最終リハーサル。「FINISH!!」で締め。前半の「機能試験」・中盤の「カード診断」と違い、現場で実際にやる操作を順番に流すシナリオ。各操作の型：制御 → 状態安定化待ち → 復旧 → 前後状態確認。
流れ：カード使用状態確認 → とまり木CH制御 → キャリア制御 → 送信アンテナ系制御(アンテナ数:2向け) → セル工事中規制 → 局舎制御 → 光伝送装置ON/OFF・リセット → チルト角遠隔制御(制御→10度に戻し) → 整流装置・空調機確認 → FINISH!!
🔑 シナリオ13は「マーク読み方の総復習ログ」。ALM・リ断・ERR・制御失敗が全部出て、全部「直前に操作があったか☆注記があるか」で説明できる。

【ALM】カード制御(RRE/LRE閉塞)で発生

カードを意図的に閉塞しています。閉塞＝強制アラーム状態にする操作なので当然出ます。閉塞解除で復旧します。

【リ断】カード制御の直後に発生

カードを閉塞するとそのカードが担っているCPRIリンクが一時的に切れます。閉塞解除で戻ります。「閉塞したら上のリンクも切れる」という依存関係がここでも出ています。

【ERR】閉塞でカードがエラー状態

閉塞操作でカードがエラー状態になります。閉塞解除すれば消えます。早見表の「ERRはリセット/復旧で消える」がそのまま出ています。

【制御失敗】局舎制御の箇所 ★注記あり

局舎制御で制御失敗が出ますが、直前の☆行にこう書いてあります。

「OF-AMPは非対応のため『制御失敗』となります。」

OF-AMP(光増幅器)はこの制御に対応していない機器なので、制御失敗が出るのは設計どおり。マークだけ見ると「失敗した」に見えるが、☆注記に理由があれば想定内。
